Content Security Policy (CSP): Una Guía Completa

Content Security Policy (CSP) es un mecanismo de seguridad HTTP que permite a los desarrolladores web controlar los recursos que el navegador puede cargar para una página web dada. Esto ayuda a mitigar los riesgos de ataques de cross-site scripting (XSS) y otros ataques de inyección, proporcionando una capa adicional de defensa contra vulnerabilidades comunes. En esencia, CSP define una lista blanca de fuentes permitidas, bloqueando cualquier recurso que no se encuentre en esa lista.

¿Cómo Funciona CSP?

CSP funciona mediante la adición de un encabezado HTTP, Content-Security-Policy, a la respuesta del servidor. Este encabezado contiene una directiva que especifica qué recursos se permiten cargar. El navegador luego utiliza esta directiva para verificar cada recurso antes de cargarlo. Si un recurso no coincide con la directiva CSP, el navegador bloqueará ese recurso.

Por ejemplo, una directiva simple podría ser: Content-Security-Policy: default-src 'self'. Esto indica que solo se permiten recursos del mismo origen (mismo protocolo, dominio y puerto).

Directivas CSP Comunes

Existen varias directivas CSP que permiten un control granular sobre los recursos cargados. Algunas de las más comunes son:

  • default-src: Define la política por defecto para todos los tipos de recursos si no se especifica otra directiva.
  • script-src: Especifica las fuentes permitidas para scripts.
  • style-src: Especifica las fuentes permitidas para hojas de estilo.
  • img-src: Especifica las fuentes permitidas para imágenes.
  • font-src: Especifica las fuentes permitidas para fuentes.
  • object-src: Especifica las fuentes permitidas para plugins (como Flash).
  • media-src: Especifica las fuentes permitidas para audio y video.
  • frame-src: Especifica las fuentes permitidas para iframes.
  • connect-src: Especifica las fuentes permitidas para solicitudes de red (fetch, XMLHttpRequest, WebSockets).
  • base-uri: Especifica las fuentes permitidas para la etiqueta <base>.
  • form-action: Especifica las fuentes permitidas para la acción de formularios.

Implementando CSP

Implementar CSP es relativamente sencillo. Se puede hacer a través de un encabezado HTTP o mediante una metaetiqueta HTML. Sin embargo, se recomienda usar el encabezado HTTP para mayor seguridad y confiabilidad.

Usando el encabezado HTTP

Content-Security-Policy: default-src 'self'; script-src 'self' 'unsafe-inline'; img-src 'self' data:;

Este ejemplo permite recursos del mismo origen, scripts del mismo origen y scripts inline ('unsafe-inline' - se debe evitar si es posible), e imágenes del mismo origen y datos en línea (data:).

Usando la metaetiqueta HTML

Aunque menos recomendado, se puede usar una metaetiqueta HTML como alternativa, aunque esto solo funciona para navegadores que soportan esta etiqueta:

<meta http-equiv="Content-Security-Policy" content="default-src 'self'; script-src 'self' 'unsafe-inline'; img-src 'self' data:;">

Nota: El uso de 'unsafe-inline' y 'unsafe-eval' debe evitarse siempre que sea posible, ya que debilitan significativamente la seguridad proporcionada por CSP.

Ventajas y Desventajas de CSP

Ventajas

  • Protección contra XSS: CSP es una defensa efectiva contra ataques XSS, previniendo la ejecución de scripts maliciosos.
  • Mayor seguridad: Añade una capa adicional de seguridad a tu aplicación web, reduciendo la superficie de ataque.
  • Prevención de clickjacking: Con una configuración adecuada, CSP puede ayudar a prevenir ataques de clickjacking.
  • Mejor postura de seguridad: CSP refuerza la seguridad general de tu aplicación web.

Desventajas

  • Complejidad de configuración: Configurar CSP correctamente puede ser complejo, requiriendo un profundo conocimiento de las directivas y sus implicaciones.
  • Posible ruptura de funcionalidad: Una configuración incorrecta de CSP puede causar que partes de tu sitio web dejen de funcionar correctamente.
  • Depuración compleja: Diagnosticar problemas causados por una configuración incorrecta de CSP puede ser difícil.

Consejos para Implementar CSP

  1. Comienza con una política restrictiva: Empieza con una política que bloquee la mayoría de los recursos y luego relaja gradualmente las restricciones según sea necesario.
  2. Prueba exhaustivamente: Prueba tu política CSP minuciosamente para asegurarte de que no rompa la funcionalidad de tu sitio web.
  3. Utiliza herramientas de auditoría: Existen herramientas que pueden ayudarte a auditar tu sitio web y detectar posibles problemas de configuración de CSP.
  4. Monitoriza los informes de violaciones: Configura los informes de violaciones para que te notifiquen cuando se produzcan intentos de cargar recursos bloqueados.
  5. Documenta tu política: Mantén una documentación clara de tu política CSP para facilitar el mantenimiento y la depuración.

Conclusión

Content Security Policy es una herramienta poderosa para mejorar la seguridad de tu aplicación web. Aunque requiere una configuración cuidadosa y pruebas exhaustivas, los beneficios en términos de protección contra ataques XSS y otras vulnerabilidades justifican el esfuerzo. Recuerda comenzar con una política restrictiva, probarla rigurosamente y monitorizar los informes de violaciones para mantener la seguridad de tu sitio web.
